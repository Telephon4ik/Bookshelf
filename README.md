## Анализ ситуации и контекста

### Ключевые вызовы
Любители книг и исследователи сталкиваются с проблемами:
- **Хаотичное хранение информации**: Книги разбросаны по заметкам, файлам Excel, закладкам в браузере.
- **Ручной ввод метаданных**: При добавлении книги приходится вручную вводить название, автора, ISBN, жанр — это отнимает 3-5 минут на книгу.
- **Отсутствие аналитики**: Невозможно быстро оценить прогресс чтения за месяц или распределение книг по жанрам.
- **Нет гибкой организации**: Статические папки не позволяют отнести одну книгу к нескольким категориям одновременно (например, "Фантастика" и "Классика").

### Влияние проблемы
- Пользователь теряет мотивацию вести учёт из-за трудоёмкости.
- Потеря времени на дублирующий ввод данных.
- Невозможность принять обоснованное решение о покупке новой книги (непонятно, сколько уже прочитано в этом жанре).

### Контекст
Проблема возникает в процессе "от покупки до прочтения и анализа". Текущий процесс (без системы) включает разрозненные инструменты:
`Покупка книги` → `Запись в блокнот/Excel` → `Чтение` → `Ручной подсчёт статистики` → `Забывание о книге`

## Обзор существующих решений

| Решение | Преимущества | Недостатки |
| :--- | :--- | :--- |
| **Google Sheets / Excel** | Доступность, гибкость | Нет автоматического поиска метаданных, нет визуальной статистики, нет статусов чтения |
| **Goodreads** | Большая база книг, социальные функции | Нет гибкой системы тегов (только полки), нет локального хранения, зависимость от интернета |
| **Calibre** | Мощный менеджер для электронных книг | Тяжёлый, неудобный для трекинга прогресса, нет веб-интерфейса (только десктоп) |
| **Notion (шаблоны)** | Красивые дашборды, гибкость | Нужно настраивать вручную, нет интеграции с внешними API, медленный при большом количестве книг |

### SWOT-анализ текущего состояния (отсутствие системы)

| | **Положительные** | **Отрицательные** |
| :--- | :--- | :--- |
| **Внутренние факторы** | **S (Сильные стороны):** Нет привязки к одному устройству, данные можно перенести. | **W (Слабые стороны):** Ручной ввод, отсутствие автоматизации, нет единого хранилища, нет аналитики. |
| **Внешние факторы** | **O (Возможности):** Появление открытых API (Open Library, Google Books) позволяет автоматизировать ввод. | **T (Угрозы):** Пользователь бросает ведение учёта из-за рутины. |

### Выбор подхода (Bookshelf)
**Аргументация:** Создание собственного лёгкого веб-приложения на Spring Boot позволяет:
1.  **Автоматизировать** 90% ввода данных через интеграцию с API.
2.  **Организовать** книги через гибкие теги (виртуальные полки).
3.  **Визуализировать** прогресс с помощью графиков (Chart.js).
4.  **Работать локально** (H2) или развернуть в сети.

## Концепция решения

### Как работает решение
Пользователь взаимодействует с единым веб-интерфейсом, где все действия интуитивны:
1.  **Поиск книги**: Вводит ISBN (или сканирует камерой) → приложение стучится к API → возвращает заполненную форму.
2.  **Организация**: Присваивает книге теги, статус ("В планах"/"Читаю"/"Прочитано").
3.  **Трекинг**: Вносит ежедневный прогресс (страницы).
4.  **Анализ**: Система строит графики и отчёты.

### Основные компоненты (MVP)
1.  **Frontend (Thymeleaf + CSS)**: Полка книг, форма добавления/редактирования, страница статистики.
2.  **Controller (Spring MVC)**: Принимает запросы, вызывает сервисы.
3.  **Service Layer**: Бизнес-логика, нормализация данных (ISBN, авторы), вызов внешних API.
4.  **Repository (JPA/Hibernate)**: Работа с БД H2.
5.  **External Integrations**: Адаптеры для Google Books API и Open Library.

### Ключевые преимущества
- **"Сканируй и получай"** : ISBN сканер + API Open Library.
- **Гибкая категоризация**: Система тегов, одна книга может быть на нескольких "виртуальных полках".
- **Автонормализация**: `"лев толстой"` → `"Лев Толстой"`, очистка ISBN.
- **Оффлайн-готовность**: Локальная БД H2, HTTPS через self-signed.

### Технические характеристики

| Компонент | Технология | Версия | Назначение |
| :--- | :--- | :--- | :--- |
| **Backend Framework** | Spring Boot | 3.1.5 | Каркас приложения (MVC, DI) |
| **ORM** | Spring Data JPA (Hibernate) | 6.x | Работа с БД |
| **Database** | H2 | 2.x | Файловая БД (локальное хранение) |
| **Template Engine** | Thymeleaf | 3.1.x | Генерация HTML-страниц |
| **API Clients** | RestTemplate | Spring | Запросы к Google Books / Open Library |
| **JSON Processing** | Jackson | 2.15 | Парсинг ответов API |
| **Frontend** | HTML5, CSS3, JS | ES6 | Пользовательский интерфейс |
| **Charts** | Chart.js | 4.4.0 | Визуализация статистики |
| **Barcode Scanner** | QuaggaJS | 0.12.1 | Сканирование ISBN через камеру |
| **Build Tool** | Maven | 3.9.x | Сборка и управление зависимостями |
| **Java Version** | OpenJDK | 17 | Среда выполнения |

### Ресурсные затраты (минимальные)
- **RAM**: 256-512 MB (для JAR + H2).
- **CPU**: 1 vCore.
- **Disk**: ~50 MB (приложение + БД на 1000 книг).
- **Network**: Требуется только для вызовов внешних API (Google Books, Open Library).
